# @shopify/shopify-app-template-react-router

## 2026.06.19
- CDO Program: **checkout is now BLOCKED while any referral-code validation is unresolved (`checkout-ui-code` extension).** Previously the extension validated codes and surfaced messages but never stopped the buyer — an order could still be placed with an invalid/foreign code present (and a `bound_other` referral-link code was only auto-removed, with no hard stop if removal failed). `Checkout.jsx` now registers a **`useBuyerJourneyIntercept`** that prevents the buyer from advancing past the current step — including the final **Pay** step, so the order is never created — whenever a single render-derived gate (`referralBlock`) is non-null. The gate covers every referral failure mode: (1) an **external/applied code still being validated** (`showValidating` — referral link or Shopify's native discount box); (2) an **invalid code that couldn't be auto-removed** (`referralBlockMessage` — set when `applyDiscountCodeChange removeDiscountCode` fails, so a known-bad code stuck on the order hard-blocks until the buyer clears it); and (3) an **invalid manually-entered code** left in the Verify box (cleared by editing/emptying the field). **`validateLinkCode` was widened from binding-only to all reasons:** any `!valid` result — unknown/inactive code (`not_found` → "Invalid Referral Code"), missing practitioner (`practitioner_missing` → "Practitioner does not exist"), or binding conflict (`bound_other` → "You are already associated with another practitioner") — now removes the code + clears the `cdo_practitioner_code` attribute and, if removal fails, raises `referralBlockMessage` to hard-block; a valid code clears the block. The referral-link effect **no longer requires buyer identity** (code-validity checks run for guests too; the permanent-binding check stays identity-gated server-side). **Capability:** added **`block_progress = true`** to `shopify.extension.toml` — without it Shopify silently downgrades `behavior:'block'` to `allow` (auto-granted in local dev; the merchant toggles it under "Checkout behavior"). When the merchant disallows blocking, `canBlockProgress` is false and the extension degrades gracefully — it still renders the reason banner (critical) with a **Remove code** action. New render state: a critical "blocked" banner for the stuck-code case; `isApplied` now excludes `referralBlockMessage` so a bad code never shows as "applied". Reuses the existing `POST /api/cdo/checkout-validate-code` (`{ code, email?, customerId? }`) — no backend/schema change. **Requires** `shopify app deploy` (extension rebuild + capability registration). **Limitation unchanged:** a fully guest checkout with no PCD email can't have its *binding* enforced at the extension (validity still is), but order ingest re-enforces server-side.
- CDO Program: **referral-link enforcement at the checkout extension (completes the permanent-binding story).** A referral link (`https://<shop>/discount/<code>`) auto-applies the practitioner's discount code to the checkout *before* the `checkout-ui-code` extension mounts; the extension previously showed any such code as "applied" with no binding check. `Checkout.jsx` now detects an **externally-applied** practitioner code (one we didn't apply this session — from a referral link or Shopify's native discount box) and validates it against the patient's permanent binding via `POST /api/cdo/checkout-validate-code` (same endpoint as manual entry, passing `email`/`customerId`). On a **`bound_other`** conflict it **removes the discount code** (`applyDiscountCodeChange removeDiscountCode` + clears the `cdo_practitioner_code` cart attribute) and shows **"You are already associated with another practitioner"** — the existing relationship is never overwritten by a foreign link. A code matching the bound practitioner, or a brand-new patient with no binding yet, is left applied (the mapping is created/kept at order ingest). New UI states: a "Validating your referral link…" interim banner (flash-free — gated by a render-derived `linkPending` + async `linkChecking`) and the rejection message rendered in the default form so the patient can still enter their own practitioner's code. The saved-code auto-apply effect now early-returns when an external code is present so the two paths don't race. Extension-only change (reuses last cut's `resolvePractitionerReferral`/`checkPatientBinding` + the order-ingest guard); requires `shopify app deploy`. **Caveat unchanged:** a fully anonymous guest (no PCD-L2 email, not logged in) can't be checked at the extension — order ingest still attributes server-side to the bound practitioner.
- CDO Program: **permanent patient↔practitioner referral relationship — enforced at registration, referral-link/order attribution, and checkout.** Once a patient (by email and/or Shopify customer id) is attributed to a practitioner, that relationship is permanent: they may afterwards only use referral codes belonging to the **same** practitioner. **New shared helpers in `services/cdo/cdo.service.js`:** `resolvePatientPractitioner({ email, customerId })` resolves the bound practitioner from `cdo_applications.referral.practitionerId` (primary, by email **or** customerId) then `cdo_referrals` (fallback, earliest row by `referredEmail` wins — the first attribution is the permanent one); `checkPatientBinding({ email, customerId, practitionerId })` returns `{ ok }` — `ok:true` when there's no binding yet (first attribution) or the candidate code's practitioner matches the bound one, `ok:false reason:"bound_other"` when it's a different practitioner. The binding compares **practitioner, not code**, so the assigned practitioner can rotate/issue new codes and the patient can use any of them without breaking the relationship. **Enforcement points:** (1) **Checkout** — `POST /api/cdo/checkout-validate-code` now accepts `{ code, email?, customerId? }` and returns the specific `result.message`: **"Invalid Referral Code"** (unknown/inactive code), **"Practitioner does not exist"** (code's practitioner no longer eligible — via `resolvePractitionerReferral`), or **"You are already associated with another practitioner"** (binding mismatch). (2) **Registration** — `POST /api/signup-form` runs `checkPatientBinding` after verifying the code and rejects a different practitioner's code with the same message (a code from the already-assigned practitioner is allowed). (3) **Order/referral-link attribution** — `resolveOrderReferral` already treats the `cdo_applications` mapping as primary; added a binding guard in the code-fallback path so a foreign code carried on an order can't re-attribute a patient who's already bound (covers the `cdo_referrals`-only case), leaving the order unattributed rather than crediting another practitioner. **Checkout extension** (`extensions/checkout-ui-code`): `ApiService.verifyCode(code, identity)` now sends the buyer email (`buyerIdentity.email`, PCD-L2 best-effort) + customer id (`customer.id`, PCD-L1 reliable); `Checkout.jsx` surfaces the backend `result.message` on the Verify step (instead of a generic "Code not found"). **Limitations:** the checkout-time block depends on the extension being placed/visible and on identity being readable (a fully guest checkout with no PCD-L2 email can't be gated *at the extension*, but order ingest still attributes server-side to the bound practitioner); a foreign code pasted into Shopify's **native** discount box bypasses the extension — fully blocking that would need a checkout/cart-validation Shopify Function (future). **Requires** `shopify app deploy` (extension rebuild) + the ns-retail app running for the new endpoint behaviour. No schema migration. Spec: `docs/payout.md` (CDO referral binding).

- CDO Program: `cdo_applications` is now the **primary source of truth** for order referral validation + practitioner mapping (`resolveOrderReferral`). The order flow first honors the buyer's existing (non-rejected) `cdo_applications` referral mapping; only when none exists does it fall back to validating the discovered code against the `cdo_practitioner_codes` catalogue (first-touch), creating the mapping. Orders that resolve to neither are treated as standard retail (no referral/commission records).
