## 2026.07.22 (fix practitioner code tag ↔ active_code divergence — the "code assigned but not applying" bug)
- **Root cause of an assigned practitioner code not auto-applying / being declined at checkout: the `code:<code>` customer TAG (read by the storefront auto-apply) and the `cdo.active_code` METAFIELD (enforced by the practitioner-discount Function) could drift out of sync.** They were written by **separate best-effort calls** — so a partial failure, or a caller that wrote only one, left the block applying a code the Function then declined (silent no-discount). **Fix:** new single consistent writer `app/utils/patientCode.js` → `syncPatientCode(shop, customerGid, code, { practitionerEmail })`. It writes the enforcement **metafield first**, then the **tag**, and on a tag-write failure **reverts the metafield to its previous value** — so the two can never be left in a harmful `tag ≠ active_code` state (first-touch with no prior value degrades safely to "no tag → auto-apply does nothing", never a wrong code). **Callers consolidated:** (1) `api/cdo-internal/assign-patient-code.js` (the portal "Assign discount code" — was two independent try/catch writes) now calls `syncPatientCode` and **surfaces a `shopifySync`/`warning` in its response** so a partial write is no longer reported as silent success; (2) `api/cdo/update-customer-referral.js` **previously wrote only the tag** (never `active_code`) — a guaranteed divergence — now writes both via `syncPatientCode`. Left as-is: the `orders/create` webhook + `checkout-apply-code` (they set both at first-touch where the prior value is null, so no harmful divergence, and altering ingestion/enforcement semantics was out of scope). **Also added** a read-only diagnostic `scripts/check-patient-code.js` (`npm run diagnose:patient-code -- --email=<patient>`) that prints the customer's `code:` tag, `cdo.active_code`, `cdo.practitioner_id`, and the matching Shopify discount's status + `cdo/config` (code/practitionerId/percentage), with a verdict flagging a tag↔metafield mismatch — the fast way to confirm this class of issue. eslint + `npm run build` clean. **Requires redeploy** for the endpoint changes to take effect; existing already-diverged customers are healed on their next assign/referral-update (or fix one immediately by re-assigning the code in the portal). Spec: this entry.

## 2026.07.22 (auto-create + durably store the dropship QBO Vendor — no hardcoded id)

## 2026.07.17
- Retail QBO invoices: **fixed tax never appearing on the customer invoice.** Reported: the retail QBO invoice showed products + shipping but no Tax row. Root cause: `retailQbo.service.createInvoiceForOrder`/`buildInvoiceLines` marked EVERY invoice line `TaxCodeRef: { value: "NON" }` (non-taxable) and passed the Shopify tax only as a bare `TxnTaxDetail: { TotalTax }` with no `TxnTaxCodeRef`. The retail QBO company uses the manual US sales-tax model (`Preferences.TaxPrefs.UsingSalesTax: true` + `TaxGroupCodeRef`, NOT Automated Sales Tax), and for that model QBO **recomputes tax to $0** when no line is taxable and no transaction-level tax code is supplied — so the passed `TotalTax` was silently dropped. Verified empirically against the retail QBO **sandbox** (create → read-back → delete): the old payload (`NON` lines + bare `TotalTax:8`) came back `TotalTax:0`; marking the product line `TAX` + adding `TxnTaxDetail.TxnTaxCodeRef` came back with the tax honored EXACTLY as passed (5→5, 31→31), independent of QBO's own configured rate — so the invoice total still equals what the customer paid in Shopify and the payment fully settles (no leftover balance). An explicit `TaxLine` was deliberately NOT sent — QBO ignores it and recomputes from the rate (confirmed in the same tests). **Fix (3 changes, all in `services/retailQbo/`):** (1) `buildInvoiceLines` marks product lines `TaxCodeRef: { value: "TAX" }` when `pricing.totalTax > 0` (else `NON` as before); shipping/discount/adjustment stay `NON` (the transaction-level `TotalTax` override is authoritative, so their codes only affect QBO's ignored recompute). (2) New cached, best-effort resolver `resolveRetailSalesTaxCodeId()` — reads the company's default sales-tax code from `Preferences.TaxPrefs.TaxGroupCodeRef`, overridable via new env `QBO_RETAIL_TAX_CODE_ID`; never throws (tax must not break invoicing). (3) `createInvoiceForOrder` sets `TxnTaxDetail: { TxnTaxCodeRef: { value: <resolved code> }, TotalTax: tax }` when the order has tax (falls back to a bare `TotalTax` if no code resolves — no worse than before). End-to-end verified through the real `createInvoiceForOrder` path with a $300 + $10.30 shipping + $31 tax order: QBO returned `TotalAmt: 341.30`, `TotalTax: 31`, Balance 341.30, product line `TAX` / shipping `NON`. **Forward-only** — existing invoices are not backfilled; re-test with a new taxed order. Config: new optional `QBO_RETAIL_TAX_CODE_ID`. `eslint` + `react-router build` clean.

- CDO Program: referral codes are now also resolved from **Shopify customer tags** (`CODE:<code>` / `REFERRAL:<code>`) when the order payload carries no code — the order webhook fetches the customer's tags via the Admin API as a fallback after the note-attribute and discount-code sources. Recorded as `attribution.source = "customer_tag"`.
- CDO Program: `cdo_applications` is now the **primary source of truth** for order referral validation + practitioner mapping (`resolveOrderReferral`). The order flow first honors the buyer's existing (non-rejected) `cdo_applications` referral mapping; only when none exists does it fall back to validating the discovered code against the `cdo_practitioner_codes` catalogue (first-touch), creating the mapping. Orders that resolve to neither are treated as standard retail (no referral/commission records).
